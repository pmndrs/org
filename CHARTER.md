# Poimandres Charter

**Status: Draft**

## 1. Mission

Poimandres is an open source developer collective for creative technology and developer tooling. We make things for people who make things because we believe code can be a craft, a creative medium, and a source of joy. Our projects are sustained through shared effort, mutual support, and trust.

## 2. Core Principles

**Simplicity.**  Software should be simple to understand and use with clear models and elegant interfaces. The API and philosophy should fit in a single readme and explained in three steps or less.

**Pragmatism.** Projects should solve demonstrated problems for a meaningful community of developers, avoiding speculative projects.

**Stability.** Software should earn long term trust. Libraries refine over time, but aim to be stable, achieved primarily through being both simple and practical.

**Quality**. Projects should meet a high standard of craft. This includes implementation, documentation, design, and the overall developer experience.

## 3. Membership

Membership is a working role within the collective. It is granted to contributors who need trusted access or take responsibility for the collective. Members are expected to uphold the collective’s mission and principles, use their access responsibly and participate in decision making.

<details>
<summary><strong>What are working roles?</strong></summary>

There is no strict definition of a working role, but common examples include project leads, maintainers, and people responsible for infrastructure or administration. The needs of the collective evolve over time and so does what it means to be a member.

</details>

## 4. Decision-Making

Decisions should be made openly and as close as possible to the work they affect. Project leads and maintainers retain autonomy over their projects, including project status.

Collective decisions use **lazy consensus**. A proposal must be presented in a designated collective communication channel and is accepted if no member objects within five days. An objection pauses the proposal until it is resolved.

<details>
<summary><strong>What are collective decisions?</strong></summary>

Anything that reasonably affects the collective at a whole is considered a collective decision. This includes adopting projects, adopting initiatives and allocating collective resources.

</details>

If a project no longer has an active lead, the collective may make decisions for it.

## **5. Adopting Projects**

Projects the collective adopts must adhere to the collective's mission and principles. A project that satisfies some but not all can still be adopted if members agree.

<details>
<summary><strong>Example criteria</strong></summary>

An adopted project should have a clearly defined purpose and fit into one of the collective's initiatives. Here are some example questions to test if a project fits in the collective.

- **Simplicity:** Is the project’s purpose simple and clear enough to explain in a couple sentences?
- **Pragmatism:** Does the project solve a real problem shared by developers beyond a single use case, product, or sponsor?
- **Stability:** Is the project’s API simple and settled enough to support long-term use?
- **Quality:** Does the project stand out in some way (API, depth, design, etc.)?

</details>

## **6. Initiatives**

The collective may adopt initiatives to set priorities, allocate collective resources, and align work toward the collective’s mission. An initiative has a definite goal and should not be intended to last indefinitely. An initiative must have a lead dev to be accepted.

<details>
<summary><strong>Example Initiatives</strong></summary>

A good initiative should define its **motivations**, **goals**, **scope of work** and the **resources** it requires. It should stand as a document that can be shared to communicate the work we are doing and how someone can contribute.

Example initiatives can be about making major releases for projects, infrastructure changes within the collective, supporting new use cases for users or tooling.

</details>

## **7. Project Policy**

Projects must have its status clearly marked. The status communicates the collective's commitment to it and expectations for future updates and can be one of the following:

- **Preview.** The project is in-development and not stable yet. Should come with a roadmap.
- **Supported.** The project is in active development with support.
- **Maintenance.** No new features. Updates are accepted to fix bugs or address security risks.
- **Archived.** No longer accepting PRs.
